# Internationalization (i18n) with Vue I18n

**Status**: ✅ Implemented (January 2026)

Diff Voyager uses [Vue I18n](https://vue-i18n.intlify.dev/) for internationalization support with English and Polish translations.

## Dependencies

- `vue-i18n@^10.0.7` - Vue 3 internationalization plugin

## Why Vue I18n?

- **Single source of truth**: All UI strings defined in locale files, not scattered in components
- **Multi-language support**: Currently supports English (en) and Polish (pl)
- **Type-safe**: Full TypeScript support with autocomplete
- **Better UX**: Users can switch languages without redeployment
- **Maintainability**: Easy to update UI text without touching component code

## CRITICAL RULE: NO HARDCODED STRINGS

All user-facing text MUST use translation keys via the `t()` function. **NEVER** hardcode strings directly in Vue components.

### ❌ WRONG - Hardcoded strings

```vue
<template>
  <NButton>Create Project</NButton>
  <NCard title="Quick Actions">
    <p>Loading...</p>
  </NCard>
</template>
```

### ✅ CORRECT - Using translation keys

```vue
<script setup lang="ts">
import { useI18n } from 'vue-i18n';

const { t } = useI18n();
</script>

<template>
  <NButton>{{ t('projects.create') }}</NButton>
  <NCard :title="t('dashboard.quickActions')">
    <p>{{ t('common.loading') }}</p>
  </NCard>
</template>
```

## Translation Structure

Translation files are located in `packages/frontend/src/i18n/locales/`:
- `en.json` - English translations (primary)
- `pl.json` - Polish translations

```
common.*           - Common UI elements (loading, save, cancel, etc.)
nav.*              - Navigation items
dashboard.*        - Dashboard-specific strings
projects.*         - Project management strings
  ├─ form.*        - Project form fields
  ├─ crawl.*       - Crawl settings
  ├─ profile.*     - Run profiles
  └─ viewport.*    - Viewport settings
runs.*             - Run management strings
  ├─ status.*      - Run status values
  └─ form.*        - Run form fields
pages.*            - Page-related strings
diffs.*            - Diff-related strings
rules.*            - Mute rules strings
  └─ form.*        - Rule form fields
settings.*         - Settings strings
errors.*           - Error messages
validation.*       - Validation messages
```

## Usage Examples

### 1. Basic translation in template

```vue
<template>
  <h1>{{ t('dashboard.welcome') }}</h1>
  <p>{{ t('dashboard.subtitle') }}</p>
</template>
```

### 2. Dynamic prop values

```vue
<template>
  <NCard :title="t('dashboard.statistics')">
    <NStatistic :label="t('dashboard.totalProjects')" :value="count" />
  </NCard>
</template>
```

### 3. Using in computed properties (for reactive data)

```vue
<script setup lang="ts">
import { computed } from 'vue';
import { useI18n } from 'vue-i18n';

const { t } = useI18n();

const viewportPresets = computed(() => [
  { label: t('projects.viewport.presets.desktop'), value: { width: 1920, height: 1080 } },
  { label: t('projects.viewport.presets.laptop'), value: { width: 1366, height: 768 } },
  { label: t('projects.viewport.presets.tablet'), value: { width: 768, height: 1024 } },
  { label: t('projects.viewport.presets.mobile'), value: { width: 375, height: 667 } },
]);
</script>

<template>
  <NSelect :options="viewportPresets" />
</template>
```

### 4. Translation with parameters (if needed in future)

```vue
<template>
  <!-- Translation file: { "message": "Showing {count} items" } -->
  <p>{{ t('pagination.showing', { count: 5 }) }}</p>
</template>
```

## Adding New Translations

When adding new features:

1. **Add translation keys to BOTH locale files** (`en.json` and `pl.json`)
2. **Use descriptive, hierarchical keys** following existing structure
3. **Never hardcode strings** in components - always use `t()` function

```json
// en.json
{
  "myFeature": {
    "title": "My Feature",
    "description": "Feature description",
    "action": "Do Something"
  }
}

// pl.json
{
  "myFeature": {
    "title": "Moja funkcja",
    "description": "Opis funkcji",
    "action": "Zrób coś"
  }
}
```

## Testing with i18n

In component tests, use the actual translation keys instead of hardcoded strings:

```typescript
import { mount } from '@vue/test-utils';
import { createI18n } from 'vue-i18n';
import en from '@/i18n/locales/en.json';

const i18n = createI18n({
  legacy: false,
  locale: 'en',
  messages: { en },
});

const wrapper = mount(MyComponent, {
  global: {
    plugins: [i18n],
  },
});

// Use translation values from locale files
expect(wrapper.text()).toContain(en.dashboard.welcome);
```

## Important Notes

- All UI text must be translatable - no exceptions
- The `t()` function is reactive and will update when language changes
- For Naive UI component props (`:title`, `:label`, `:description`), use `:propName="t('key')"`
- Keep translation keys organized and follow the existing hierarchy
- Update both `en.json` and `pl.json` simultaneously when adding new keys
